## 1. 本地算力部署说明

### 1.1 部署目标与资源约束

项目采用单机 NVIDIA DGX Spark 作为本地推理与智能体运行节点。设备提供 128 GB CPU+GPU 统一内存，可减少 CPU 与 GPU 之间的数据搬运，并支持模型权重、KV Cache、智能体编排进程及工具服务共享本地内存资源。所有核心推理均在本地完成，可降低外部网络依赖并增强数据闭环能力。

资源约束：Qwen3.6-35B-A3B 的 BF16 权重占用约 70 GB，长上下文场景下 KV Cache 约需 30~40 GB；Step3.7-Flash 的原始 BF16 权重占用超过 400 GB。若不进行压缩，前者几乎占满可用内存，后者无法直接装载，因此必须先量化，再为智能体编排、工具调用和系统运行预留空间。

### 1.2 智能体与模型服务部署流程

1. **资源规划与模型选择。** 根据任务复杂度、上下文长度和响应时延要求选择本地模型。Qwen3.6-35B-A3B 作为高吞吐主模型；Step3.7-Flash 可用于需要更强推理能力、且可接受更高时延的任务。
2. **生成量化模型制品。** 离线执行校准与量化，产出 TensorRT-LLM、vLLM/compressed-tensors 或 llama.cpp 可直接加载的模型文件，避免在在线服务启动阶段重复量化。
3. **启动本地推理服务。** 按模型格式选择 TensorRT-LLM、vLLM 或 llama.cpp 引擎，限制最大上下文、并发数和 KV Cache 占用，并将推理能力封装为仅在本机或内网开放的统一接口。
4. **部署智能体编排服务。** 智能体服务与模型服务同机部署，通过本地接口完成规划、工具选择和结果生成；工具服务、业务数据与日志按权限隔离，避免模型进程直接访问不必要的系统资源。
5. **验证并上线。** 在固定提示词、上下文长度和并发配置下验证 TTFT、TPOT、输出吞吐、峰值内存与任务正确率；达到性能和精度阈值后固化模型制品、启动参数和版本清单。

### 1.3 本地模型路由与内存策略

- 优先使用 Qwen3.6-35B-A3B 处理通用问答、工具调用和高频交互，以获得较低时延和较高输出吞吐。
- 在复杂推理场景下按需路由至 Step3.7-Flash，避免两套大模型同时常驻导致统一内存不足。
- 采用单并发基线控制峰值内存，并根据实际上下文长度设置 KV Cache 上限；智能体编排与工具进程使用剩余内存，必要时对低频模型按需加载。
- 模型权重、量化参数和激活 scale 均随部署制品固化；运行参数由配置文件统一管理，以便复现测试结果和快速回滚。

## 2. 大模型优化说明

### 2.1 Qwen3.6-35B-A3B 优化路径

项目采取“先压缩到可部署体积，再针对解码密集型场景加速”的递进式方案。各阶段均以单并发测试为基线，并在速度、显存/统一内存占用与模型能力之间进行权衡。

| 层级 | 瓶颈判断 | 核心动作 | 优化理由 | 输出吞吐 | TTFT | TPOT |
| --- | --- | --- | --- | ---: | ---: | ---: |
| L0 | 权重+KV Cache 挤占统一内存 | 拆分权重、KV Cache 与智能体内存预算 | 建立可部署目标 | — | — | — |
| L1 | 模型体积与权重读取带宽 | MoE NVFP4；敏感层 BF16；静态 scale；TensorRT-LLM | 先压参数主体，保住敏感层 | 60+ tok/s | 314.46 ms | 14.75 ms |
| L2 | 量化制品与 vLLM 后端不匹配 | compressed-tensors；敏感层 FP8；W4A4 dynamic | 匹配后端并增强激活鲁棒性 | 72+ tok/s | 302.95 ms | 13.69 ms |
| L3 | 敏感层仍占用较高精度路径 | AWQ 压缩敏感层，形成全 4-bit 主体 | 激活感知保护重要通道 | 96+ tok/s | 293.63 ms | 10.53 ms |
| L4 | 自回归 decode 串行依赖 | MTP1/MTP2 投机解码 | 停止继续降位宽，转向解码并行 | 99+/100+ tok/s | — | 9.82/8.38 ms |

> **测试口径：** concurrency=1；TTFT/TPOT 为长输入场景实测值，MTP 吞吐为长输出场景实测值。“+”表示达到该数值以上。不同提示词长度、采样参数、MTP 接受率和软件版本会影响结果。

### 2.2 量化后的能力保持

在 MMLU-Pro、GPQA 和 AIME 2025 三项基准上，激进量化方案相对 BF16 的分数变化约为 0.01~0.4，整体处于项目可接受范围。建议在正式提交或复测时同步保存测试脚本、采样参数与模型版本，以确保结果可复现。

| 方案 | MMLU-Pro | GPQA | AIME 2025 |
| --- | ---: | ---: | ---: |
| BF16 | 85.75 | 86.36 | 92.50 |
| FP8 | 85.75 | 86.74 | 93.12 |
| NVIDIA | 85.60 | 87.12 | 91.88 |
| Unsloth | 85.85 | 86.74 | 92.29 |
| 本项目激进量化 | 85.58 | 87.75 | 91.67 |

### 2.3 Step3.7-Flash 优化路径

Step3.7-Flash 的 BF16 权重占用超过 400 GB，无法直接装载到 128 GB 统一内存。项目采用 llama.cpp 兼容的 Q4_K_S 4-bit 量化，将模型文件压缩至约 100~110 GB，在精度损失可控的前提下实现单机部署。

| 方案 | 运行引擎 | 模型体积 | 输出吞吐 | TTFT | TPOT |
| --- | --- | --- | ---: | ---: | ---: |
| Q4_K_S 量化 | llama.cpp | 约 100~110 GB | 26.89 tok/s | 2172.29 ms | 37.98 ms |
| 原生 MTP 投机解码 | llama.cpp + MTP | 沿用 Q4_K_S 制品 | 40+ tok/s（长输出） | — | — |

## 4. 系统级推理优化方案

### 4.1 算子与执行图优化

- **按阶段识别瓶颈。** 使用 Roofline 与运行时剖析区分 Prefill 的计算密集特征和 Decode 的访存/启动瓶颈；Prefill 侧优先大块 GEMM 与 IO-Aware Attention，Decode 侧评估小 M GEMM/GEMV、Split-K 和 Persistent Kernel。低精度格式仅在模型、引擎和硬件共同支持时启用。
- **优化 MoE 与 Attention。** 评估 Router、Top-K、Dispatch、Combine 融合及 Grouped GEMM；长上下文根据模型结构采用 Paged Attention，并在具备 GQA/MQA 时复用 KV。目标是减少中间写回、小 Kernel 启动和专家负载不均。
- **约束式融合与离线调优。** 优先使用 TensorRT-LLM 已验证 Kernel；只有剖析确认热点后，才在 cuBLASLt、CUTLASS、Triton 或自定义 CUDA Kernel 间选择。调优结果按设备、模型、Shape 和精度缓存，避免在线搜索抬高时延。
- **CUDA Graph 与安全回退。** 对稳定的 Decode Shape 进行分桶并捕获执行图，动态 MoE 路由尽量保留在设备端；未覆盖 Shape 必须回退到标准 CUDA 路径，并记录回退原因，防止静默降速。

### 4.2 连续批处理与资源感知调度

- **Continuous Batching。** 在生产多请求场景采用 Token Block 级连续批处理、Token Budget、Decode 保留预算与 Chunked Prefill，减少长 Prefill 对 Decode 的阻塞。现有 concurrency=1 数据继续作为单请求基线，不与多并发结果直接比较。
- **Shape Bucketing。** 预定义常用 Batch/上下文分桶、预分配工作区并复用 CUDA Graph，请求映射到可容纳它的最小 Bucket，以降低动态分配、编译与同步开销。
- **准入与服务等级。** 请求执行前估算权重、KV Cache 与工作区峰值；使用带老化的加权公平队列，在容量不足时优先保护交互请求，并提前拒绝或降级可能触发 OOM 的任务。
- **多模型驻留。** Qwen3.6-35B-A3B 与 Step3.7-Flash 不应在当前 128 GB 统一内存中同时常驻。高频模型常驻、低频模型按需加载；切换时限制 Prefill、预取权重并预热 Kernel，避免统一内存页迁移干扰 Decode。

### 4.3 KV Cache 全生命周期管理

- **分页与共享。** 采用 KV Page、Page Table 和预分配内存池，使用引用计数与 Copy-on-Write 支持前缀复用和并行采样。FP8/INT8 KV 仅在目标模型和引擎验证精度与 Kernel 支持后启用。
- **Prefix Cache。** 缓存键至少包含 Token ID、模型/量化配置、适配器和权限域；结合 Radix Tree 或 Block Hash、只读共享页和成本感知 LRU，避免跨配置错误复用。
- **分层缓存边界。** DGX Spark 为 CPU+GPU 统一内存，将数据“转移到 CPU”不能扩展物理容量。L0/L1 仅表示统一内存内的热/冷优先级；真正的容量外溢应进入 NVMe（L2）或选择重计算（L3），并通过异步 I/O、双缓冲和显式预取避免阻塞 Decode。
- **长期运行治理。** 监控页面利用率、连续块、分配失败和统一内存迁移；通过在线回收、增量一致性检查以及 Supervisor–Worker 滚动轮换释放 CUDA Context 与分配器碎片。

### 4.4 双机并行与 Prefill/Decode 分离

- **通信基线。** 使用 NCCL 承载集合通信、UCX/RDMA 传输 KV，校验拓扑、带宽和 RDMA 是否发生回退，并将计算、集合通信和 KV 传输放入独立流以尝试重叠。
- **TP=2。** 适用于量化后仍超过单机容量或单节点计算成本过高的模型；采用行/列并行及 All-Reduce、Reduce-Scatter、All-Gather，并对 Attention Head 和 MoE TP/EP 进行匹配。不能据此推断 Step3.7-Flash 的原始 BF16 权重可在两台设备上直接部署。
- **Prefill/Decode 分离。** 仅在 Prefill 干扰明显且 RDMA 传输成本低于隔离收益时启用。Decode 端预留 KV Page 并管理 Prefix Cache，使用 Credit/Backpressure 控制发送，避免接收空间不足。
- **模式与故障恢复。** TP=2 与 PD 分离分别固化为部署模板，按容量、负载和通信成本选择；切换时排空请求并预热，通过部署签名、超时、幂等请求标识和输出边界保证一致性。

## 5. 实施优先级与验收

| 优先级 | 实施范围 | 关键内容 | 验收重点 |
| --- | --- | --- | --- |
| P0 | 保持现有量化与单并发基线 | 固化模型校验值、量化配置、软件版本和启动参数 | 复现实测 TTFT、TPOT、tok/s 与能力基准 |
| P1 | 单机低风险优化 | 性能剖析、准入控制、内存池、监控、稳定 Shape 的 CUDA Graph | 无精度回退；峰值内存、P95/P99 时延和 OOM 率改善 |
| P2 | 单机服务化优化 | Continuous Batching、Chunked Prefill、Prefix Cache、Shape Bucketing、可选 KV 量化 | 在目标并发下提升吞吐，同时守住 TTFT/TPOT 与能力阈值 |
| P3 | 双机扩展 | TP=2 或 Prefill/Decode 分离，NCCL/UCX/RDMA 与故障回退 | 通信无静默降级；故障可恢复；双 Rank 状态一致 |
